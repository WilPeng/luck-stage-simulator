# 公演评分体系说明

## 一、选手个人得分

### 计算公式

```
属性分 = round(vocal × 归一化声乐权重 + dance × 归一化舞蹈权重 + charm × 归一化魅力权重)

难度系数 = 1 - (歌曲难度 - 1) × 0.1
  · 难度 1 → 1.0（无衰减）
  · 难度 5 → 0.6（衰减 40%）

发挥加分 = 发挥值 × 2
  · 发挥值范围: -10 ~ 20
  · 发挥加分范围: -20 ~ 40

原始分 = 属性分 × 难度系数 + 发挥加分

最终得分 = clamp(原始分, 0, 120)  （截断到 0~120 之间）
```

### 评级映射

| 得分范围 | 评级 | 描述 |
|---------|------|------|
| ≥ 85    | S    | 完美舞台 |
| ≥ 65    | A    | 出色表现 |
| ≥ 45    | B    | 稳定发挥 |
| ≥ 25    | C    | 略有不足 |
| < 25    | D    | 失误较多 |

### 关键参数

| 参数 | 说明 | 来源 |
|------|------|------|
| 选手属性 `vocal/dance/charm` | 训练获得，无上限 | `player.attributes` |
| 歌曲权重 `vocalWeight/danceWeight/charmWeight` | 管理员在歌曲库中配置 | `song.*Weight` |
| 归一化权重 | 原始权重 ÷ 三项权重之和 | 确保三项之和 = 1.0 |
| 歌曲难度 `difficulty` | 1~5，管理员在歌曲库中配置 | `song.difficulty` |
| 发挥值 `performanceValue` | 选手抽取 or 管理员代生成，范围 -10~20 | `PerformanceValue` 表 |

## 二、团队得分 & 最终票数

### 团队得分

```
团队得分 = round(全队成员个人最终得分的平均值)
```

不再有随机加成。

### 最终票数

```
最终票数 = 500 + 团队得分 × 3 + 团队平均魅力值

团队平均魅力值 = round(全队成员 charm 属性之和 ÷ 成员数)
```

### 票数构成拆解

| 组成部分 | 公式 | 说明 |
|---------|------|------|
| 基础票数 | 500 | 固定值 |
| 团队得分加成 | 团队得分 × 3 | 团队综合实力的体现 |
| 平均魅力加成 | 团队平均 charm | 魅力属性高的队伍获得额外加成 |

### 最终排名

所有队伍按 `最终票数` 降序排列，票数相同时名次并列。

## 三、个人喜爱度权重

### 总权重公式

```
总权重 = ①基础属性贡献 + ②实时发挥贡献 + ③团队排名加成 + ④队内MVP加成 + ⑤观众缘随机值
```

### 分项明细

#### ① 基础属性贡献

```
基础属性贡献 = round(charm × 2)
```

- 只依赖 `charm`（魅力）属性
- charm 越高此项得分越高，无上限

#### ② 实时发挥贡献

```
实时发挥贡献 = max(0, 发挥值 + random(-5, 5))
```

- 基于选手在公演中的实际发挥值
- 叠加 ±5 的随机波动
- 最低为 0（不会为负）

#### ③ 团队排名加成（等差递减）

```
团队排名加成 = round(30 × (总队伍数 - 队伍排名) / (总队伍数 - 1))
```

| 排名（5队示例） | 加成 |
|:---:|:---:|
| 第 1 名 | +30 |
| 第 2 名 | +23 |
| 第 3 名 | +15 |
| 第 4 名 | +8 |
| 第 5 名 | +0 |

- 第 1 名固定 +30
- 最后一名固定 +0
- 中间队伍按等差数列均匀分配

#### ④ 队内 MVP 加成

```
队内MVP加成 = 队内排名第1 ? random(10, 20) : 0
```

- 仅队内个人得分第 1 名的选手获得
- 10~20 之间的随机值

#### ⑤ 观众缘随机值

```
观众缘随机值 = random(0, 15)
```

- 纯随机，模拟观众缘的偶然性

### 差距拉大调整

在计算完原始权重后、进行加权投票抽样前，执行一次调整：

1. 找出**权重最低**的选手，设其权重为 `minWeight`
2. 如果 `minWeight > 10`，计算 `N = minWeight - 10`
3. **所有选手的权重同时减去 N**
4. 权重最低的选手变为 10，其余选手也同等幅度降低

**目的**：拉开权重差距，使权重高的选手获得更多投票。

### 投票机制

| 参数 | 值 |
|------|-----|
| 评审人数 | 1000 人 |
| 每人投票数 | 3 票 |
| 抽样方式 | 加权不放回抽样 |
| 总票数 | 3000 票 |

- 每位评审按照调整后的权重进行加权不放回抽样
- 权重越高的选手被选中的概率越大
- 每次选中后从候选池移出，同一评审不会重复投票给同一人

## 四、数据流程图

```
训练阶段               公演阶段                   淘汰阶段
─────────           ──────────                ──────────
选手训练抽卡          ① 选手抽取发挥值               进入淘汰阶段后
积累属性值            (-10~20)                   展示个人喜爱度排名
                    ↓
                    ② 管理员公演结算
                    ├─ 计算个人得分 (属性分×难度+发挥加分)
                    ├─ 计算团队得分 (个人分均值)
                    └─ 计算最终票数 (500+团队分×3+avgCharm)
                    ↓
                    ③ 管理员生成大众评审投票
                    ├─ 计算个人喜爱度权重 (5项之和)
                    ├─ 差距拉大调整
                    └─ 加权不放回抽样 (1000评审×3票)
                    ↓
                    ④ 管理员释放最终结果
                    ├─ 选手端展示评审席 (Phase 4)
                    └─ 淘汰阶段展示权重排名 (Phase 5)
```
